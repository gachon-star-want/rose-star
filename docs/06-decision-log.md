## 해결된 grill-me 질문

Q001. 최종 실행 대상 플랫폼을 **Windows PE `.exe`**로 확정할 것인가?

결정: **예.**

이유:

- 공모전 심사자가 실행하기 가장 무난한 형태다.
- 단일 exe로 제출 조건을 설명하기 쉽다.
- Win32 API만 쓰면 외부 런타임 없이 가능하다.
- DOS보다 현대 실행 환경 리스크가 낮다.
- macOS/Linux보다 심사 환경 가정이 보수적이다.

Q002. Windows PE 아키텍처를 **x86 32비트 exe**로 시작할 것인가?

결정: **예.**

이유:

- `x86 PE`: 바이너리가 더 작을 가능성이 있고 Win32 자료가 많다.
- `x64 PE`: 현대 Windows 네이티브 환경에 더 직접적이다.

현재 결정은 **x86 PE로 프로토타입을 먼저 만들고, 실행 환경 리스크가 보이면 x64 PE를 백업으로 비교**하는 것이다.

Q003. 어셈블러를 **FASM**으로 확정할 것인가?

결정: **예.**

이유:

- PE 실행 파일을 직접 만들기 쉽다.
- 작은 단일 파일 어셈블리 프로젝트와 잘 맞는다.
- 링커 설정 부담이 NASM 계열보다 적다.
- Win32 예제가 많아 초기 창/메시지 루프를 빠르게 만들 수 있다.

Q004. 내부 해상도를 **256x144**로 확정할 것인가?

결정: **예.**

이유:

- 16:9 비율이라 현대 화면에 확대하기 쉽다.
- 프레임버퍼가 36,864픽셀이라 직접 렌더링 비용이 낮다.
- 16x16 타일 기준 16x9 타일 화면이 되어 레벨 설계가 단순하다.
- 작은 스프라이트와 제한된 팔레트가 공모전 미학에 잘 맞는다.

Q005. 오디오는 1차 목표에 **효과음만 포함하고, 음악은 2차 목표**로 둘 것인가?

결정: **예.**

이유:

- 1차 완성은 게임 루프와 전투 피드백이 먼저다.
- 효과음은 설치/공격/피해/클리어 피드백에 직접 기여한다.
- 음악까지 1차에 넣으면 waveOut, 타이밍, 패턴 데이터, 믹싱 부담이 커진다.
- 음악은 게임 루프가 완성된 뒤 남는 바이트를 채우는 exact-fill 컨텐츠로 쓰기 좋다.

Q006. 최종 스테이지 수를 **32개 마이크로 섹터**로 둘 것인가?

결정: **당시 예였으나, 이후 64개 스테이지로 상향 대체.**

이유:

- 초대 Mario는 8월드 x 4스테이지, 총 32스테이지 구조였다.
- 1.44MB는 40KiB Mario ROM의 36배에 해당한다.
- 용량상 32섹터는 충분히 가능하다.
- 각 섹터를 짧고 밀도 높은 마이크로 스테이지로 만들면 레벨 디자인 부담을 관리할 수 있다.
- 이후 타워디펜스 피벗과 1.44MB 여유를 반영해 64개 스테이지로 올렸다.

Q007. 스테이지를 **오브젝트 명령 기반 레벨 데이터**로 저장할 것인가?

결정: **예.**

이유:

- Mario식으로 전체 타일맵을 그대로 저장하지 않고 "어디에 경로/설치 영역/웨이브를 놓는다"는 명령으로 저장하면 매우 작다.
- 64개 스테이지를 넣어도 데이터 크기를 작게 유지할 수 있다.
- 같은 명령을 조합해 많은 변주를 만들 수 있다.
- 레벨 에디터 없이도 텍스트/배열 기반으로 설계하기 좋다.

Q007-A. 장르를 **64스테이지 마이크로 타워디펜스**로 피벗할 것인가?

결정: **예.**

이유:

- 64개 스테이지 목표와 잘 맞는다.
- 정교한 점프 물리보다 구현 리스크가 낮다.
- 고정 경로와 웨이브 데이터는 오브젝트 명령 기반 저장과 잘 맞는다.
- 작은 타워/적 조합으로 많은 변주를 만들 수 있다.

Q008. 타워디펜스 경로 모델을 **고정 경로**로 확정할 것인가?

결정: **예.**

이유:

- 자유 배치/동적 길찾기는 구현 복잡도가 크게 오른다.
- 고정 경로만으로도 64개 스테이지 변주가 충분하다.
- 경로 데이터를 작게 저장할 수 있다.
- 레벨 하네스에서 경로 연결성과 웨이브를 검증하기 쉽다.

Q009. 타워/적 종류를 **1차 타워 3종/적 4종, 최종 타워 5종/적 8종**으로 잡을 것인가?

결정: **예.**

이유:

- 타워 3종이면 단일/강공격/범위 역할이 분리된다.
- 최종 타워는 단일/강공격/범위/슬로우/지원으로 역할을 분리한다.
- 범위와 슬로우를 분리하면 레벨 디자인에서 순차 해금과 조합 학습이 명확해진다.

- Endless는 기존 맵·웨이브 자산 + 스케일 공식만으로 가볍게 구현한다. Random은 D013/D028 명령 포맷(Header/PATH/BUILD/WAVE)을 런타임에 생성하는 작은 생성기(경로 자기교차·연결성 검증 포함)로 구현하며, 1.44MB exact-fill 보너스 콘텐츠로 적합하다.

확정 세부:

- 해금 판정 기준은 **해당 스테이지 Easy 클리어**다. (24번째를 Easy로만 깨도 Endless 해금, 48번째를 Easy로만 깨도 Random 해금. 접근성 우선.)
- 모드 정의 확정: **Endless = 고정 경로 맵 + 무한 웨이브(적 주기 스케일링, 생존 웨이브 수가 점수)**, **Random = 절차적 생성 맵(경로·빌드존 랜덤) + 웨이브**. 절차적 생성기는 Random 모드 한 곳에만 격리한다.

Q029. 화면 표시 사양을 어떻게 확정할 것인가? (D008 재검토)

결정: **4:3 / 256x192 / 창 모드 전용 / 정수배 4배(1024x768).** (상세 D046)

이유:

- 사용자가 풀스크린 확대 시 픽셀이 과하게 커 보이는 점을 우려 → 원인은 "비율"이 아니라 "풀스크린"이므로 **창 모드 전용**으로 원천 해소한다.
- 256x144(16:9)만 16x16 타일에 깔끔히 떨어진다는 것은 오해였다. **256x192(4:3)도 16x12로 동일하게 떨어진다.**
- 진짜 판별 기준은 비율이 아니라 고정경로 여부다. 고정경로 TD는 가로가 표준이라 16:9도 정당하지만, **4:3은 세로 스위치백 경로 여지가 더 크고** 레트로 CRT 감성에 부합한다. 사용자는 레트로 외형 + 현대적 내용 톤을 원해 4:3을 선택.
- 1:1 정사각형은 가로 흐름을 죽여 제외. Share Card(1080x1080)는 별도 스냅샷이라 플레이 비율과 일치할 필요 없다.
- 표시 블록(비율+해상도+창모드+정수배)을 한 결정으로 묶어 락하고 재오픈하지 않는다.
