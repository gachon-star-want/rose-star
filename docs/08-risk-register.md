# Risk Register

## R001: 어셈블리어 개발 난이도

영향: 높음  
가능성: 높음

설명:

전체 게임을 어셈블리어로 만들면 작은 바이너리는 얻을 수 있지만 개발 속도가 느려진다. 특히 창 생성, 오디오, 웨이브/타워 상태, 디버깅에서 시간이 든다.

대응:

- 1차 목표를 고정 경로 타워디펜스로 제한
- 타워 1종, 적 1종, 첫 해넘이 몇 개를 첫 마일스톤으로 둠
- 최종 44개 해넘이는 짧은 마이크로 해넘이로 제한
- 디버그/하네스를 초기에 마련

## R002: 대상 플랫폼 불확실

영향: 높음  
가능성: 중간

설명:

공모전 페이지는 독립 실행 파일을 요구하지만 대상 OS를 명시하지 않는다. 심사자가 실행 가능한 환경을 가정해야 한다.

대응:

- Windows 단일 exe를 추천안으로 둠
- 제출 전 주최자에게 권장 OS를 확인
- README에 실행 환경을 명확히 적음

## R003: 개발 머신과 대상 실행 환경 차이

영향: 중간  
가능성: 높음

설명:

현재 개발 머신은 macOS로 보인다. Windows exe를 만들면 로컬 실행 검증이 바로 어렵다.

대응:

- Wine 또는 Windows VM/실기기 검증 루트 확보
- 최소한 바이너리 크기/구조 검증은 로컬에서 수행
- smoke 검증은 Windows 환경에서 주기적으로 실행

## R003-A: x86 PE 호환성 가정

영향: 중간  
가능성: 낮음

설명:

Windows x86 PE는 일반적인 x64 Windows에서 WOW64로 실행된다. 그러나 주최 측이 특수한 64비트 전용 정책을 두거나, 테스트 환경이 32비트 앱 실행을 제한하면 문제가 될 수 있다.

대응:

- 주최 측 공지에서 대상 OS/아키텍처를 확인
- x86 PE를 1차 타깃으로 개발
- 필요 시 x64 PE 포트를 백업으로 유지
- 플랫폼 레이어를 분리해 아키텍처 전환 비용을 낮춤

## R004: 웨이브 밸런스 부족

영향: 높음  
가능성: 중간

설명:

타워디펜스는 그래픽보다 웨이브 밸런스와 배치 판단이 핵심이다. 숫자만 늘려 어려워지면 44개 해넘이가 지루해진다.

대응:

- Phase 3~5에서 웨이브/타워 수치 기록
- 각 스테이지의 학습 목표를 하나로 제한
- 스테이지 제작 전에 1차 타워 3종과 1차 적 4종의 기본 수치를 고정
- Hard Mode 기준으로 클리어 가능성을 먼저 확인
- Easy Mode 초당 지급량은 완화용으로만 조절

## R005: 과한 기능 욕심

영향: 높음  
가능성: 높음

설명:

어셈블리어 프로젝트에서 동적 길찾기, 복잡한 업그레이드, 보스, 저장, 연출, 음악, 맵 에디터까지 넣으려 하면 완성 가능성이 낮아진다.

대응:

- 최소 완성 기준을 명시
- 1차 범위: 커서, 고정 경로, 타워 1종, 적 1종, 8개 스테이지
- 2차 범위: 최종 타워 5종, 최종 적 8종, 효과음, 추가 연출
- 동적 길찾기는 비목표로 둠

## R006: 원문 규칙 변경 또는 모순

영향: 중간  
가능성: 중간

설명:

현재 페이지에는 마감 시간과 상금 정보에 모순이 있다. 심사 기준도 작성 중이다.

대응:

- 더 보수적인 기준을 내부 기준으로 사용
- 주기적으로 원문 재확인
- 제출 직전 최신 공지 확인

## R007: 최종 제출물 메타데이터

영향: 중간  
가능성: 중간

설명:

macOS에서 패키징하면 `.DS_Store` 같은 파일이 제출 폴더에 섞일 수 있다. 압축 해제 후 용량과 구성에 영향을 줄 수 있다.

대응:

- `dist/`를 깨끗하게 생성
- 숨김 파일 제거
- 최종 제출 폴더 크기 직접 측정
- 압축 후 다른 폴더에 풀어서 재측정

## R008: Exact Fill 목표가 릴리즈를 복잡하게 만듦

영향: 중간  
가능성: 높음

설명:

최종 제출물을 정확히 1,474,560바이트로 맞추는 것은 공모전 규칙보다 더 엄격한 자체 조건이다. 무의미한 패딩으로 해결하면 컨셉은 지킬 수 있어도 "1바이트도 안 날린다"는 제품 원칙에는 약하다.

대응:

- 개발 중에는 1,400,000바이트 이하 안전선을 유지
- 릴리즈 단계에서 남는 바이트를 자동 계산
- 남는 공간은 숨겨진 레벨, 리플레이, 사운드 패턴, 메타데이터로 우선 채움
- 마지막 수십 바이트만 체크섬/릴리즈 서명/정렬 데이터로 처리

## R009: 온라인 리더보드 범위 폭증

영향: 중간  
가능성: 중간

설명:

온라인 리더보드는 서버, 네트워크, 계정/닉네임, 부정 기록 검증, 장애 처리를 요구한다. 단일 exe 1.44MB 게임의 핵심 범위를 벗어날 수 있다.

대응:

- 온라인 리더보드는 1차 목표에서 제외
- 공유 가능한 기록 코드와 공유 카드 이미지를 우선
- 기록 코드에 체크섬을 포함해 최소한의 검증 가능성 확보
