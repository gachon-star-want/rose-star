# Build And Harness

## 목적

1.44MB 공모전에서는 "게임이 된다"와 "작다"를 동시에 검증해야 한다. 하네스는 개발 중 반복 확인을 자동화하기 위한 장치다.

## 검증 계층

| 계층 | 검증 내용 |
| --- | --- |
| Build | 어셈블리 소스가 실행 파일로 빌드되는가 |
| Size | 제출 폴더가 1,474,560바이트 이하인가 |
| Dependency | 외부 런타임이나 누락 파일이 없는가 |
| Smoke | 실행 직후 창이 뜨고 프레임이 진행되는가 |
| Determinism | 같은 입력 리플레이가 같은 상태를 만드는가 |
| Level | 레벨이 깨질 수 없는 구조가 아닌가 |
| Release | 최종 패키지가 원문 제출 조건을 만족하는가 |

## 현재 로컬 전제

현재 로컬에는 `clang`만 확인되었다. `fasm`, `nasm`, `wine`은 없다. 어셈블러는 FASM으로 확정했다.

FASM 확보 방식은 **프로젝트 로컬 고정 툴체인 + 버전/해시 기록**으로 한다. 시스템에 설치된 `fasm`은 편의상 사용할 수 있지만, 공식 빌드 하네스는 사용 중인 FASM 경로, 버전, SHA256을 확인해야 한다.

현재 대상 실행 파일은 **Windows x86 PE `.exe`**로 잡는다. 로컬 macOS에서는 직접 실행 검증이 어려우므로, Windows 실기기/VM/Wine 중 하나의 검증 경로를 확보해야 한다.

## 추천 디렉터리 구조

```text
.
├─ docs/
├─ src/
│  ├─ main.asm
│  ├─ platform_win32.asm
│  ├─ render.asm
│  ├─ game.asm
│  └─ data.asm
├─ tools/
│  ├─ toolchain/
│  │  ├─ README.md
│  │  └─ fasm/
│  ├─ size-check.sh
│  ├─ package.sh
│  ├─ level-check.py
│  └─ replay-check.sh
├─ build/
└─ dist/
```

최종 제출물에는 `src/`, `tools/`, `docs/`를 넣지 않는다. 제출 대상은 `dist/`의 실행 파일과 꼭 필요한 파일만이다.

## Toolchain 하네스

목표:

- FASM 버전 고정
- FASM 바이너리 SHA256 기록
- 빌드에 사용된 실제 FASM 경로 출력
- 시스템 전역 설치물과 프로젝트 로컬 툴체인 혼동 방지

권장 검증 명령 후보:

```text
make toolchain-check
```

기대 동작:

1. 프로젝트 로컬 FASM 경로 확인
2. FASM 실행 가능 여부 확인
3. 버전 문자열 또는 바이너리 식별 정보 출력
4. SHA256을 문서화된 값과 비교
5. 불일치 시 빌드 실패

## Build 하네스

표준 빌드 명령은 Makefile을 공개 인터페이스로 둔다. 복잡한 로직은 `tools/` 스크립트로 분리한다.

목표 빌드 명령:

```text
make build
```

권장 공개 명령:

```text
make toolchain-check
make build
make size
make smoke
make level-check
make replay-check
make package
make release-check
```

Makefile 원칙:

- Makefile은 얇게 유지한다.
- FASM 호출, 용량 계산, 패키징, exact-fill, replay 검증은 `tools/` 스크립트로 위임한다.
- 최종 제출물에는 Makefile과 `tools/`를 넣지 않는다.

기대 동작:

1. `build/` 생성
2. FASM으로 `src/main.asm` 빌드
3. `dist/`에 실행 파일 복사
4. 용량 측정
5. 실패 시 명확한 에러 출력

## Size 하네스

필수 검증:

```text
dist total bytes <= 1474560
dist total bytes <= 1400000  # 개발 중 안전 상한
dist total bytes == 1474560  # 릴리즈 exact-fill 모드
```

측정 방식:

- 제출 폴더 전체의 실제 파일 크기를 합산한다.
- macOS 리소스 포크, `.DS_Store`, 숨김 메타데이터가 섞이지 않도록 한다.
- 최종 zip 크기가 아니라 압축 해제 후 크기를 기준으로 한다.

릴리즈 모드에서는 `1,474,560바이트 이하`가 아니라 `정확히 1,474,560바이트`를 요구한다. 남는 바이트는 `tools/exact-fill` 계열 도구가 계산하고, 빌드 시스템은 남는 크기만큼 의미 있는 데이터 섹션을 생성해야 한다.

권장 exact-fill 흐름:

```text
1. release exe 생성
2. dist/ 정리
3. 현재 제출 폴더 크기 측정
4. remaining = 1474560 - current_size 계산
5. remaining이 음수면 실패
6. remaining만큼 secret/metadata/fill data 생성
7. 최종 제출 폴더 크기 재측정
8. 정확히 1474560이 아니면 실패
```

## Dependency 하네스

목표:

- OS 기본 DLL/API 외의 파일이 필요한지 확인한다.
- 제출 폴더 안에 실행에 필요한 모든 비 OS 파일이 있는지 확인한다.

Windows 대상이면 최종적으로 실제 Windows 환경에서 다음을 확인한다.

- 깨끗한 폴더에 exe만 두고 실행
- 인터넷 없이 실행
- 관리자 권한 없이 실행
- 한글/공백 없는 경로와 있는 경로 모두 확인

## Smoke 하네스

최소 smoke 기준:

- 실행 후 2초 안에 창 생성
- 256x192 내부 프레임버퍼가 화면에 확대 출력
- 프레임 카운터 증가
- 입력을 받으면 커서 위치 변화
- 타워 설치 입력 후 타워가 화면에 표시
- `Esc`로 종료

개발 빌드에는 다음 디버그 모드를 둘 수 있다.

```text
--test-smoke
--replay replay/basic.bin
--frames 600
--checksum
```

최종 릴리즈 빌드에서는 명령행 옵션과 디버그 문자열을 제거할 수 있다.

## Deterministic Replay

타워디펜스 밸런스는 수동 테스트만으로 충분하지 않다. 최소한 한 가지 입력 리플레이를 둔다.

리플레이 데이터 예:

```text
frame, buttons
0, RIGHT
4, RIGHT
8, DOWN
12, PLACE
30, NEXT_TOWER
36, PLACE
60, START_WAVE
```

검증 값:

- 600프레임 후 커서 위치
- 현재 달/해넘이 번호
- 남은 생명/바이트
- 보유 자원
- 살아 있는 적 수
- 설치된 타워 수
- 상태 체크섬

## Level 하네스

레벨 데이터 검증 항목:

- 경로 시작점이 존재하는가
- 경로 도착점이 존재하는가
- 경로가 시작점에서 도착점까지 연결되는가
- 경로가 고정 경로 명령만으로 재현되는가
- 설치 가능 칸이 경로와 겹치지 않는가
- 경로 위에 타워를 설치할 수 없게 되어 있는가
- 최소 1개 이상의 웨이브가 존재하는가
- 시작 자원으로 최소 1개 타워를 설치할 수 있는가
- Hard Mode에서도 클리어 가능하도록 설계/수동 검증되었는가
- Easy Mode의 초당 지급량이 스테이지 예산을 망가뜨리지 않는가
- 타일 ID가 범위를 벗어나지 않는가
- 웨이브 적 타입이 범위를 벗어나지 않는가

자동으로 완전한 클리어 가능성을 증명하기는 어렵다. 대신 명백한 데이터 오류와 불가능한 경제 설정을 잡는 데 집중한다.

## 실패 기준

아래 중 하나라도 해당하면 릴리즈 불가다.

- 제출 폴더가 1,474,560바이트 초과
- 릴리즈 exact-fill 빌드가 1,474,560바이트 미만
- 남는 공간을 의미 없이 방치
- 브라우저나 별도 런타임 설치가 필요
- 실행 직후 크래시
- 키보드 없이 진행 불가한 화면에서 멈춤
- 클리어 불가능한 스테이지 존재
