# Size Budget

## 총량

공식 상한: **1,474,560바이트**

개발 중 안전 상한: **1,400,000바이트**

개발 중 여유분: **74,560바이트 이상**

최종 릴리즈 목표: **정확히 1,474,560바이트**

이 프로젝트는 "작게 만들기"만 목표로 하지 않는다. 최종 제출물은 3.5인치 HD 플로피 한 장의 실제 물리 용량을 정확히 채워야 한다. 즉, 초과도 실패지만 미달도 제품 목표 관점에서는 실패다.

## Mario Size Benchmark

초대 Super Mario Bros.는 8월드 x 4스테이지, 총 32스테이지 구조다. NES판 기준으로 흔히 말하는 원본 ROM 규모는 **약 40KiB**로 잡는다.

```text
Super Mario Bros. NES ROM: 32KiB PRG + 8KiB CHR = 40KiB
1.44MB floppy target: 1,474,560 bytes
1,474,560 / 40,960 = 36
```

즉, 이 공모전의 상한은 초대 Mario의 원본 ROM 용량을 기준으로 정확히 **36배**다.

이 사실은 중요한 제품 메시지다. 최종 게임은 플랫포머가 아니라 타워디펜스지만, Mario가 보여준 "작은 데이터로 많은 스테이지를 만드는 방식"은 그대로 참고한다. 우리는 이를 **44개 마이크로 해넘이**(4달×11, D059)로 구현한다.

- NES는 타일/스프라이트/사운드를 처리하는 전용 하드웨어가 있었다.
- Mario ROM에는 NES 하드웨어, TV 출력, OS 같은 것이 들어 있지 않다.
- 우리 Windows exe는 PE 헤더, import table, 창/입력/렌더링/오디오 glue code를 포함해야 한다.
- 반대로 Windows의 GDI/오디오/입력 API는 OS가 제공하므로 우리가 전부 구현할 필요는 없다.

결론: 용량상 44해넘이는 충분히 가능하다(64여도 여유였다). Easy/Hard 모드는 맵 데이터를 복제하지 않고 초당 별빛 지급 여부로 갈라지므로, 체감상 88개 챌린지를 제공해도 데이터 부담은 작다. 병목은 용량이 아니라 엔진 완성도, 웨이브 밸런스, 레벨 디자인 시간이다.

참고:

- NES/iNES 포맷은 PRG-ROM을 16KiB 단위, CHR-ROM을 8KiB 단위로 기록한다: https://www.nesdev.org/NESDoc.pdf
- Super Mario Bros.의 표준 ROM-only 구성은 32KiB PRG + 8KiB CHR로 정리되어 있다: https://www.retroreversing.com/super-mario-bros

## 초기 예산안

| 항목 | 목표 크기 | 비고 |
| --- | ---: | --- |
| 실행 파일 헤더/섹션 | 4~32KB | 플랫폼/링커에 따라 변동 |
| 플랫폼 코드 | 16~64KB | 창, 입력, 타이머, 오디오 |
| 렌더러 | 8~48KB | 프레임버퍼, 타일, 스프라이트 |
| 게임 로직 | 16~80KB | 타워, 적, 경로, 웨이브, 경제, 상태 |
| 레벨 데이터 | 8~64KB | 44개 해넘이, Easy/Hard는 같은 데이터 공유 |
| 그래픽 데이터 | 1~32KB | 비트패턴/절차 생성 우선 |
| 오디오 데이터 | 0~8KB | 파형 생성 우선 |
| 텍스트/폰트 | 0~16KB | 시스템 폰트 또는 초소형 비트맵 |
| 의미 있는 채움 데이터 | 남는 전부 | 숨겨진 스테이지, 리플레이, 사운드 패턴, 메타데이터 |

어셈블리어로 직접 만들면 1.44MB는 넉넉할 수 있다. 그러나 방심하면 디버그 심볼, 정적 링크, 외부 런타임, 이미지 파일 때문에 순식간에 초과할 수 있다. 반대로 너무 작게 끝나면 이 프로젝트의 "플로피 한 장을 꽉 채운다"는 약속을 못 지킨다.

따라서 개발 중에는 작게 유지하고, 릴리즈 단계에서 남는 바이트를 설계된 데이터로 채운다.

## Exact Fill 전략

최종 제출물 크기를 정확히 맞추는 방식은 우선순위를 둔다.

1. 플레이 가능한 컨텐츠 추가
2. 숨겨진 스테이지/이스터에그 추가
3. 팔레트/타일 변형 데이터 추가
4. 절차적 음악/효과음 패턴 추가
5. 리플레이/고스트 입력 데이터 추가
6. 게임 안에서 볼 수 있는 달/해넘이 메타데이터 추가
7. 체크섬/릴리즈 서명/빌드 정보 추가
8. 최후 수단으로만 패딩 사용

패딩을 쓰더라도 파일 뒤에 의미 없는 0을 붙이는 방식은 피한다. 실행 파일 또는 데이터 섹션 안에서 `FILLER`가 아니라 `LAST_SUNSET`, `SECRET_STAR`, `ROSE_LABEL`처럼 명명하고, 게임에서 접근 가능한 데이터로 만든다.

## Share Card Size Note

공유 카드 이미지는 실행 파일에 포함하지 않는다. 결과 화면에서 생성해 파일로 저장한다.

- 1080x1080 BMP는 생성 파일이 커질 수 있지만 제출물 크기에 포함되지 않는다.
- BMP writer는 구현이 작아 1차 릴리즈에 적합하다.
- PNG encoder는 보너스 목표다.
- 외부 PNG 라이브러리는 제출물 크기와 복잡도를 키우므로 사용하지 않는다.

## Exact Fill 검증 규칙

릴리즈 빌드는 다음 두 조건을 모두 만족해야 한다.

```text
dist total bytes == 1474560
dist total bytes <= 1474560
```

두 번째 조건은 공모전 규칙 검증이고, 첫 번째 조건은 이 프로젝트 자체의 제품 검증이다.

## 크기 절감 원칙

### 그래픽

- 이미지 파일보다 타일 비트패턴을 코드/데이터로 보관한다.
- 1비트 또는 2비트 타일 + 팔레트 조합을 우선한다.
- 같은 타일을 팔레트만 바꿔 여러 지역처럼 보이게 한다.
- 큰 배경 이미지는 만들지 않는다.
- 적 종류는 늘리되 적 바디 리소스는 3개 이하로 제한한다.
- 8x8/16x16 도트 타일, 팔레트 교체, 2프레임 애니메이션, 행동 플래그 조합으로 다양성을 만든다.
- Easy/Hard 시각 차이는 새 이미지가 아니라 팔레트 변형으로 처리한다.
- Hard Mode는 게임 영역 팔레트 반전/고대비 변형을 사용하되, UI 텍스트 가독성은 유지한다.

### 레벨

- 전체 타일맵을 저장하지 말고 경로/설치 영역/웨이브를 오브젝트 명령으로 저장한다.
- 반복되는 빈칸/설치 가능 영역/경로 구간은 RLE로 압축한다.
- 한 해넘이를 지나치게 길게 만들 필요는 없다. 짧은 해넘이 44개가 낫다.
- 웨이브는 적 타입, 수, 간격, 지연시간만 저장한다.

### 사운드

- `.wav`, `.mp3`, `.ogg` 파일을 넣지 않는다.
- 음계, 길이, 파형 종류만 데이터로 둔다.
- 효과음은 함수로 생성한다.

### 코드

- 표준 라이브러리 의존을 피한다.
- 디버그 출력/문자열은 최종 빌드에서 제거한다.
- 테이블 기반 로직을 활용하되 과도한 범용화는 피한다.
- 64비트보다 32비트가 작다면 32비트도 검토한다.

## 금지에 가까운 선택

- Unity/Godot/Unreal
- Electron
- 브라우저 실행 전제
- 대형 폰트 파일
- 압축 해제 후 필요한 DLL 묶음
- 고해상도 이미지 리소스
- 긴 음원 파일

## 매 빌드 체크

각 빌드마다 아래 값을 기록한다.

```text
build id:
exe size:
submit folder size:
remaining bytes to exact fill:
largest section:
asset/data size:
meaningful fill size:
notes:
```

용량은 마지막에 최적화하는 것이 아니라 개발 리듬 안에 넣는다.
